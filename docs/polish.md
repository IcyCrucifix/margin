# Stage 2 Polish — manual, nightly, and bring-your-own-AI

Stage 2 turns a lecture's source file plus your page-linked memos into one comprehensive, Obsidian-compatible polished note. The design separates **what the AI does** (write a note body) from **what the system does** (choose inputs, verify freshness, install the note, maintain links) — so the AI is a replaceable component, and everything it can affect is guarded deterministically.

## What a polish run produces

`Lecture Notes/Polished/<course>/<date> - <title> - Polished.md` containing YAML frontmatter, the H1 title, an Obsidian callout linking course / source file / raw memos — all generated by the **finalizer**, never by the AI — followed by the AI-written body. The raw note's `status` flips to `polished-available`, the library records the input hash, and the hub note is regenerated.

## The pipeline

Every run, manual or nightly, is the same sequence (`content_reader/polish.py`):

1. **Reconcile** course-folder routing and repair links ([obsidian-sync.md](obsidian-sync.md)).
2. **Skip checks** — refuse if the lecture has no memos; return *already up to date* if a current polished note exists; a per-document lock file (`runtime/jobs/<id>.lock`) prevents concurrent runs.
3. **Hash the inputs**: `input_hash = sha256(source_sha256 + page-memo JSON)`.
4. **Invoke the AI agent** with a prompt naming three read-only inputs — the original source file, the extracted page text, and the raw memo file — and two required outputs: write the note body to a draft path under `runtime/drafts/`, then run the finalizer.
5. **Finalize deterministically**: `scripts/finalize_polished_note.py --doc-id <id> --body-file <draft> --expected-hash <hash>` re-computes the hash; if any memo changed while the AI was writing, the stale draft is rejected and nothing is installed. Otherwise it wraps the body with frontmatter/links, writes atomically, updates raw-note status, library, and hub.
6. **Verify**: the run only reports success if a polished note with the current hash actually exists afterwards.

### Manual

Press **Stage 2 — Polish now** in the app. `POST /api/doc/<id>/polish` starts a background job; the UI polls `GET /api/jobs/<job>` until it completes, skips, or fails.

### Automatic (nightly)

Two headless entry points exist so any scheduler can drive the same pipeline:

```bash
python3 scripts/pending_lectures.py --json   # list lectures with missing/stale polished notes
python3 scripts/polish_pending.py            # run Stage 2 for all of them (exit 1 if any failed)
```

*Pending* means: has at least one memo, and the polished note is missing or its recorded hash no longer matches — so unchanged lectures are never re-polished, and edited memos automatically queue their lecture again.

The reference deployment schedules `polish_pending.py` as a nightly task **inside the AI agent app itself** (a Codex scheduled task, since the agent must be signed in anyway). If your agent has no scheduler, a plain `launchd`/cron job works identically — e.g. `9 3 * * * cd /path/to/margin && python3 scripts/polish_pending.py >> /tmp/margin-polish.log 2>&1`.

## Safety contract (holds for any AI)

- **Note-only.** The agent process is started with its workspace rooted at `runtime/drafts/`, with the vault as the only additional writable directory — the project source tree is read-only to it. The prompt restates this; the sandbox enforces it.
- The agent may create exactly one file itself (the draft); every vault mutation goes through the finalizer.
- The finalizer refuses drafts outside the project workspace, empty bodies, bodies that try to include frontmatter, and stale hashes.
- Sources and raw memos are never modified by a polish run.
- Prompt quality bar: preserve every substantive memo, cover the full source page-by-page in a prerequisite-first structure, Obsidian math delimiters (`$…$`/`$$…$$`) only, no invented facts.

## Bring your own AI

The built-in integration shells out to the **Codex CLI** (`codex exec --sandbox workspace-write --cd <drafts> --add-dir <vault> …`), using the signed-in app — no API key. To use a different agent, satisfy this contract:

> Given the generated prompt, the agent must (1) read the three input paths, (2) write the note body to the draft path, (3) execute the finalizer command line from the prompt, and (4) exit non-zero on failure. It must be able to write **only** under `runtime/drafts/` and the vault.

That is one command list in `run_codex_polish()` (`content_reader/polish.py`). Drop-in examples:

```python
# Claude Code (https://claude.com/claude-code)
command = [
    "claude", "-p", prompt,
    "--allowed-tools", "Read,Write,Bash",
    "--add-dir", str(store.drafts_root), "--add-dir", str(store.vault),
]

# Gemini CLI
command = ["gemini", "--sandbox", "--include-directories",
           f"{store.drafts_root},{store.vault}", "-p", prompt]
```

Keep the run directory at `store.drafts_root` and grant the narrowest write access your agent supports; the hash/finalizer guards do not depend on the agent behaving.

### No AI at all

The pipeline works with a human as the "agent":

```bash
python3 scripts/pending_lectures.py --json      # pick a doc id + input_hash
# read the source / extracted text / raw memos listed there, write body.md yourself
python3 scripts/finalize_polished_note.py --doc-id <id> \
    --body-file runtime/drafts/body.md --expected-hash <input_hash>
```

You get the same frontmatter, links, status flip, and hub update as an AI run.

## Failure modes

| Symptom | Meaning |
|---|---|
| *Add at least one page memo…* | Stage 2 needs your memos; it is not a summarizer of bare slides. |
| *Already up to date* | Inputs unchanged since the last polish — nothing rewritten. |
| *Stage 2 is already polishing this lecture* | Lock held by a running job (stale locks: delete `runtime/jobs/<id>.lock`). |
| *Raw notes changed while polishing* | You edited a memo mid-run; the stale draft was rejected. Run again. |
| *Codex is unavailable* | The configured agent CLI was not found — install it or swap the command as above. |
| *finished without installing a current polished note* | The agent exited 0 but never ran the finalizer (or the hash moved); check `runtime/jobs/<job>.log`. |
